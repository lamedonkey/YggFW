================================================================================
  YggFW — лёгкий фильтр входящих соединений для сети Yggdrasil под Windows
================================================================================

Версия: 11  |  Язык: C  |  Платформа: Windows x64
Зависимость: WinDivert (https://reqrypt.org/windivert.html)


  КОМУ ЭТО НУЖНО, А КОМУ — НЕТ
  ══════════════════════════════

Если вы работаете под Linux — смело закрывайте этот документ. На Linux описанная
ниже проблема отсутствует в принципе: подсистема netfilter корректно обрабатывает
туннельные интерфейсы, и никакой дополнительной защиты не требуется.

Если же вы под Windows и используете сеть Yggdrasil — читайте дальше.


  О ЧЁМ ВООБЩЕ РЕЧЬ
  ══════════════════

YggFW — это небольшая программа-фильтр для входящих IPv6-соединений под Windows.
Она создавалась специально для сети Yggdrasil, однако без каких-либо изменений
подойдёт для любого IPv6-интерфейса.

Программа работает как Windows-сервис в фоновом режиме, незаметно для пользователя,
и перехватывает входящие пакеты ещё до того, как они доберутся до встроенного
брандмауэра Windows.


  ПРОБЛЕМА: ПОЧЕМУ СТАНДАРТНОЙ ЗАЩИТЫ НЕ ХВАТАЕТ
  ════════════════════════════════════════════════

Сеть Yggdrasil работает через виртуальный сетевой интерфейс — программный туннель,
который выглядит для Windows не как привычная сетевая карта, а как особый
программный адаптер.

Встроенный брандмауэр Windows (Windows Defender Firewall) умеет блокировать
нежелательные входящие соединения на обычных интерфейсах — Ethernet, Wi-Fi, VPN.
Но с туннельными интерфейсами типа Yggdrasil происходит нечто неожиданное:
брандмауэр не применяет к ним политику «блокировать всё неизвестное» автоматически.
Трафик приходит через драйвер туннеля напрямую, и Windows воспринимает это как
«локальную доставку», не требующую фильтрации по умолчанию.

Итог.

Узел в сети Yggdrasil под Windows фактически открыт для входящих соединений со
всей сети — дефолтная блокировка входящих соединений к нему не применяется.
Поэтому в Windows Defender Firewall нужно вручную создать правило «блокировать всё».
Однако после такого правила невозможно создать из него исключения, задуманные для
разрешения соединений с доверенных адресов.

Казалось бы, выход прост: создать в брандмауэре правило «заблокировать всё из
Yggdrasil», а затем добавить исключения для нужных адресов. Но здесь возникает
вторая проблема. В Windows Defender Firewall блокирующие правила имеют приоритет
над разрешающими, если оба правила относятся к одному и тому же трафику. Иными
словами, широкое правило «блокировать» отменяет более узкое правило «разрешить»,
даже если то было создано раньше.

Таким образом, стандартными средствами защитить сеть Yggdrasil под Windows и
одновременно оставить доступ для доверенных адресов — не получится.


  КАК РАБОТАЕТ YggFW
  ══════════════════

Представьте, что ваш компьютер — это дом. Сеть — это улица, по которой приходят
гости. Тогда Windows Defender Firewall — это дверь вашего дома. Проблема в том,
что дверь в случае с Yggdrasil оказывается открытой по умолчанию, а замок на ней
работает непредсказуемо.

YggFW — это ворота в заборе вокруг вашего участка. Они стоят перед домом, ближе
к улице. Гость сначала должен пройти через ворота, и лишь потом — подойти к двери
дома. Если ворота закрыты — до двери он просто не доберётся.

Технически: YggFW использует библиотеку WinDivert, которая перехватывает пакеты
на уровне сетевого стека Windows ещё до того, как они попадут в Windows Defender
Firewall. Это значит, что заблокированный пакет никогда не дойдёт до брандмауэра
вообще — он будет отброшен на самом раннем этапе.

Программа поддерживает:
  - блокировку целых подсетей IPv6 (например, всего диапазона 200::/7)
  - разрешение доступа для конкретных адресов или подсетей
  - разрешение доступа к конкретным портам (TCP и UDP)
  - stateful-фильтрацию (conntrack): ответный трафик на ваши исходящие запросы
    пропускается автоматически — как и должно работать нормальное соединение
  - запись журнала в файл с ротацией по времени
  - работу в виде Windows-сервиса без участия пользователя


  УСТАНОВКА
  ═════════

1. Создайте папку C:\Bin\YggFW\ и скопируйте в неё следующие файлы:

     C:\Bin\YggFW\
     ├── yggfw.exe
     ├── WinDivert.dll
     ├── WinDivert64.sys
     ├── settings.txt      (создастся автоматически при первом запуске)
     └── rules.txt         (создастся автоматически при первом запуске)

2. Убедитесь, что у вас есть права администратора.

3. Откройте командную строку от имени администратора и выполните:

     sc create YggFW binPath= "C:\Bin\YggFW\yggfw.exe" start= auto
     sc description YggFW "Lite Yggdrasil (IPv6) Firewall"
     sc start YggFW

4. Проверьте, что сервис запущен:

     sc query YggFW

   Вы должны увидеть строку STATE : 4 RUNNING.

Для остановки и удаления сервиса:

     sc stop YggFW
     sc delete YggFW

Примечание: при первом запуске в папке C:\Bin\YggFW\ автоматически создадутся
файлы settings.txt и rules.txt с настройками по умолчанию, а также папка Logs\
для журналов.


  НАСТРОЙКИ: файл settings.txt
  ═════════════════════════════

Файл читается при каждом запуске программы. Строки, начинающиеся с символа #,
и пустые строки игнорируются. Формат: ключ=значение.

────────────────────────────────────────────────────
  ОСНОВНЫЕ ПАРАМЕТРЫ
────────────────────────────────────────────────────

loglevel = 0
  Уровень вывода на экран (консольный режим).
  0 — полная тишина. Используется когда программа работает как сервис.
  1 — показывать только заблокированные соединения.
  2 — показывать заблокированные и явно разрешённые (кроме тех, что прошли
      по умолчанию).
  3 — показывать абсолютно всё.
  4 — максимально подробный режим для отладки.
  При работе в режиме сервиса Windows loglevel принудительно устанавливается
  в 0 — сервис не имеет экрана для вывода сообщений.

response = 0
  Что делать с заблокированным пакетом.
  0 — молча выбросить. Источнику кажется, что адреса не существует.
  1 — отправить уведомление об отказе (TCP RST или ICMPv6 Unreachable).
  Значение 0 предпочтительно: оно не раскрывает информацию о вашем узле.

conntrack = 1
  Интеллектуальная (stateful) фильтрация.
  1 — включена. Программа следит за вашими исходящими соединениями и
      автоматически разрешает ответный трафик. Например, если вы обратились
      к DNS-серверу, его ответ будет пропущен автоматически.
  0 — выключена. Придётся вручную прописывать разрешения для всего ответного
      трафика, что крайне неудобно.

ct_tcp_timeout = 120
  Время жизни автоматического разрешения для TCP-соединения (в секундах).
  После закрытия соединения или истечения времени разрешение удаляется.

ct_udp_timeout = 30
  Время жизни автоматического разрешения для UDP (в секундах).

ct_icmpv6_timeout = 10
  Время жизни автоматического разрешения для ICMPv6 echo (ping) в секундах.
  Привязано к конкретному идентификатору запроса — чужой ping не пройдёт.

────────────────────────────────────────────────────
  ЖУРНАЛИРОВАНИЕ В ФАЙЛ
────────────────────────────────────────────────────

DumpLevel = 1
  Уровень записи в файл журнала. Работает независимо от loglevel.
  0 — не писать в файл.
  1 — записывать только блокировки. Оптимально для постоянной работы.
  2 — блокировки и явные разрешения.
  3 — абсолютно всё.

DumpFile = .\Logs\yggfw-%Y-%m-%d.log
  Путь к файлу журнала. Допускаются маски даты и времени в стиле strftime:
    %Y — год (2026), %m — месяц (04), %d — день (15), %H — час (21).
  Папки создаются автоматически.
  Пример для другого расположения: C:\Logs\yggfw\yggfw-%Y-%m-%d.log

DumpStart = D
  Как часто создавать новый файл журнала:
  H — каждый час
  D — каждый день (по умолчанию)
  W — каждую неделю
  M — каждый месяц
  Y — каждый год

Пример строки в журнале:
  2026-04-04 21:04:15.602 -T [316:c51a::]:80 -> [227:3f13::]:5297 [FIN][ACK] {60/0} @ DENY 200::/7 any * -> *

  Расшифровка: 
    -T  = заблокирован TCP-пакет
    +T  = разрешён TCP-пакет
    >U  = разрешён UDP-пакет через conntrack (ответный трафик)
    .I  = пропущен ICMPv6 по умолчанию
    {60/0} = размер пакета 60 байт, payload 0 байт
    @ ...  = сработавшее правило

────────────────────────────────────────────────────
  ФЛАГИ ДЕТАЛИЗАЦИИ (только при loglevel=4)
────────────────────────────────────────────────────

LogPacketNumber = 1   Нумерация пакетов в выводе
LogStats        = 1   Счётчики: всего / разрешено / заблокировано
LogMatches      = 1   Какое правило сработало
LogOutcoming    = 1   Исходящие пакеты (используются conntrack)
LogConntrack    = 1   События conntrack: добавлено / обновлено / удалено


  ПРАВИЛА ФИЛЬТРАЦИИ: файл rules.txt
  ════════════════════════════════════

Это главный файл настройки. Здесь вы описываете, кого пускать, а кого — нет.

Правила проверяются сверху вниз. Срабатывает первое совпавшее правило.
Если ни одно не подошло — пакет пропускается (разрешён по умолчанию).

────────────────────────────────────────────────────
  ФОРМАТ ПРАВИЛА
────────────────────────────────────────────────────

  <действие> <адрес_источника> <протокол> [<порт_источника> -> <порт_назначения>]

Действие (action):
  ALLOW — пропустить
  DENY  — заблокировать

Адрес источника (src_addr):
  *                          — любой адрес
  200::1                     — конкретный IPv6-адрес
  200::/7                    — диапазон адресов (подсеть)
  325:62b8:f811:b821::/64    — более узкая подсеть

Протокол (proto):
  tcp     — только TCP
  udp     — только UDP
  icmpv6  — только ICMPv6 (ping и другие). Порты для него не указываются.
  any     — TCP, UDP и ICMPv6 вместе

Порты:
  * -> *      любой порт источника, любой порт назначения
  * -> 443    любой порт источника, порт назначения 443 (HTTPS)
  53 -> *     порт источника 53 (ответ DNS-сервера), любой порт назначения

────────────────────────────────────────────────────
  ПРИМЕРЫ ПРАВИЛ
────────────────────────────────────────────────────

# Доверенный хост — разрешить всё без ограничений
ALLOW 225:62b8:f811:b821:6c6b:8fcc:c01b:425c  any  * -> *

# Доверенная подсеть — например, ваши устройства
ALLOW 325:62b8:f811:b821::/64  any  * -> *

# DNS-серверы — разрешить только их ответы (они отвечают с порта 53)
ALLOW 308:84:68:55::   udp  53 -> *
ALLOW 308:25:40:bd::   udp  53 -> *

# Разрешить входящие соединения на веб-сервер
ALLOW *  tcp  * -> 80
ALLOW *  tcp  * -> 443

# Разрешить ping (иначе вас нельзя будет пропинговать)
ALLOW *  icmpv6

# Заблокировать весь остальной трафик из Yggdrasil
DENY  200::/7   any  * -> *
DENY  300::/64  any  * -> *

────────────────────────────────────────────────────
  КАК РАБОТАЕТ CONNTRACK НА ПРАКТИКЕ
────────────────────────────────────────────────────

Допустим, у вас в rules.txt стоит только блокировка всего диапазона 200::/7.
Вы открываете браузер и заходите на сайт в сети Yggdrasil. Ваш компьютер
отправляет TCP-запрос на сервер сайта. YggFW видит этот исходящий запрос и
автоматически создаёт временное разрешение: «пропустить ответный TCP-пакет от
этого сервера на этот порт в течение 120 секунд». Когда сервер отвечает — пакет
проходит. После закрытия вкладки соединение закрывается, временное разрешение
удаляется.

Без conntrack вы бы не смогли получить никакого ответа, даже если сами
инициировали соединение.

────────────────────────────────────────────────────
  МИНИМАЛЬНАЯ КОНФИГУРАЦИЯ
────────────────────────────────────────────────────

Если вам всего лишь необходимо защитить свой компьютер от нежелательных соединений
и не нужно давать доступ для разрешённых адресов, просто добавьте в rules.txt
два правила — этого достаточно:

  DENY  200::/7   any  * -> *
  DENY  300::/64  any  * -> *

Conntrack при этом обеспечит нормальную работу исходящих соединений.
