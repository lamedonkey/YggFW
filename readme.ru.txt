
YggFW — лёгкий фильтр входящих соединений для Windows и сети Yggdrasil
========================================================================

YggFW — это небольшая, но очень полезная программа для пользователей Windows,
которые работают в сети Yggdrasil или в любой другой сети на базе протокола IPv6.
Программа перехватывает нежелательные входящие соединения и защищает ваш компьютер
от нежданных гостей из сети раньше, чем они успевают добраться до системы.

Если вы работаете под Linux — можете смело закрыть эту страницу. Описанная
здесь проблема существует только в Windows. В Linux сетевой фильтр корректно
обрабатывает туннельные интерфейсы из коробки, и никаких дополнительных средств
защиты для Yggdrasil не требуется.


Почему вообще нужна отдельная программа?
-----------------------------------------

Казалось бы, в Windows есть встроенный брандмауэр — Windows Defender Firewall.
Почему его недостаточно?

Дело вот в чём. Yggdrasil создаёт виртуальный сетевой интерфейс — не физический,
как ваша сетевая карта или Wi-Fi, а программный, туннельный. И именно здесь
кроется проблема: Windows Defender Firewall применяет политику «блокировать все
неизвестные соединения» только к обычным сетевым адаптерам. Для туннельных
интерфейсов этот механизм попросту не срабатывает — трафик поступает в систему
в обход стандартной проверки.

Итог.

Узел в сети Yggdrasil под Windows фактически открыт для входящих соединений
со всей сети — дефолтная блокировка входящих соединений к нему не применяется.
Поэтому в Windows Defender Firewall нужно вручную создать правило «блокировать
всё». Однако после такого правила невозможно создать из него исключения,
задуманные для разрешения соединений с доверенных адресов.

По этой причине разработан YggFW — инструмент, который берёт управление
в свои руки до того, как дело доходит до встроенного брандмауэра.


Как это работает?
-----------------

Представьте, что ваша сеть — это дорога, по которой едут все желающие.
Встроенный брандмауэр Windows — это дверь вашего дома: он решает, кого
впустить внутрь. Но чтобы добраться до двери, нужно сначала войти в ворота
вашего участка.

YggFW — это и есть те самые ворота. Он стоит на пути раньше брандмауэра
и разворачивает нежелательных гостей ещё на подъезде, не давая им даже
постучать в дверь дома.

Технически: YggFW работает на основе библиотеки WinDivert и перехватывает
пакеты до того, как они попадают в стек Windows. Заблокированный пакет
просто исчезает — никакого ответа источнику, никаких следов. Это называется
«тихий сброс», и именно он создаёт ощущение, что вашего адреса в сети
как будто не существует.


Установка
---------

1. Создайте папку, например: C:\Bin\YggFW\

2. Скопируйте туда файлы:

   C:\Bin\YggFW\
   ├── main.exe          <- сама программа
   ├── WinDivert.dll     <- системная библиотека
   ├── WinDivert64.sys   <- драйвер
   ├── settings.txt      <- настройки (создаётся автоматически)
   ├── rules.txt         <- правила фильтрации (создаётся автоматически)
   └── Logs\             <- папка для журналов (создаётся автоматически)

3. Откройте командную строку от имени администратора и выполните:

   sc create YggFW binPath= "C:\Bin\YggFW\main.exe" start= auto
   sc description YggFW "WinDivert IPv6 Stateful Firewall (Yggdrasil)"
   sc start YggFW

   Чтобы проверить, что сервис работает:

   sc query YggFW

4. Для остановки и удаления:

   sc stop YggFW
   sc delete YggFW

Программа может работать и как обычное приложение — просто запустите main.exe.
В этом случае вы увидите журнал событий в консольном окне.


Настройки: файл settings.txt
-----------------------------

Файл создаётся автоматически при первом запуске. Формат простой: каждая строка —
это пара «ключ=значение». Строки, начинающиеся с символа #, — это комментарии,
они игнорируются.

Основные параметры:

  loglevel = 0..4
    Определяет, насколько подробно программа отчитывается в консоли.
    0 — полная тишина (рекомендуется при работе как сервис)
    1 — показывать только заблокированные соединения
    2 — заблокированные плюс явно разрешённые
    3 — абсолютно все соединения
    4 — максимальная детализация для отладки
    По умолчанию: 0

  response = 0 или 1
    0 — заблокированный пакет молча выбрасывается. Отправитель не получает
        никакого ответа и не знает, существуете ли вы в сети вообще.
    1 — отправителю уходит отказ (TCP RST или ICMPv6 «порт недоступен»).
    По умолчанию: 0 (рекомендуется)

  conntrack = 0 или 1
    Включает «умную» фильтрацию: программа запоминает исходящие соединения
    и автоматически разрешает ответный трафик. Без этого режима вы не сможете
    открыть ни одну страницу или отправить ни один запрос — ответы будут
    заблокированы. Подробнее — в разделе про conntrack.
    По умолчанию: 1 (включено)

  ct_tcp_timeout = секунды
    Как долго ждать ответа на TCP-соединение. По умолчанию: 120

  ct_udp_timeout = секунды
    Как долго ждать ответа на UDP-запрос. По умолчанию: 30

  ct_icmpv6_timeout = секунды
    Как долго ждать ответа на ping (ICMPv6 echo). По умолчанию: 10


Параметры журнала (файл на диске):

  DumpLevel = 0..3
    Что записывать в файл журнала на диске.
    0 — ничего не записывать
    1 — только блокировки (рекомендуется для постоянной работы)
    2 — блокировки и явные разрешения
    3 — всё подряд
    По умолчанию: 1

  DumpFile = путь к файлу
    Куда сохранять журнал. Поддерживаются маски даты: %Y (год), %m (месяц),
    %d (день), %H (час). Нужные папки создаются автоматически.
    По умолчанию: .\Logs\yggfw-%Y-%m-%d.log
    Пример:       C:\Logs\yggfw\yggfw-%Y-%m-%d.log

  DumpStart = H, D, W, M или Y
    Как часто создавать новый файл журнала:
    H — каждый час
    D — каждый день
    W — каждую неделю
    M — каждый месяц
    Y — каждый год
    По умолчанию: D (новый файл каждый день)

Пример строки в журнале:

  2026-04-04 21:04:15.602 -T [316:c51a::]:80 -> [227:3f13::]:5297 [FIN][ACK] {60/0} @ DENY 200::/7 any *->*
  2026-04-04 21:04:20.192 +T [227:d351::]:12007 -> [227:3f13::]:5045 [PSH][ACK] {97/37} @ ALLOW 325::/64 any *->*

  Первый символ обозначает тип события:
    +  пакет разрешён статическим правилом
    -  пакет заблокирован
    >  пакет разрешён через conntrack (ответ на ваш исходящий запрос)
    .  пакет разрешён по умолчанию (нет подходящего правила)
  Второй символ — протокол: T (TCP), U (UDP), I (ICMPv6)


Правила фильтрации: файл rules.txt
------------------------------------

Это главный файл настройки. Именно здесь вы решаете, кого пускать, а кого нет.
Файл тоже создаётся автоматически при первом запуске.

Принцип работы прост: программа просматривает правила сверху вниз и применяет
первое подходящее. Если ни одно правило не подошло — пакет пропускается
(это называется «разрешить по умолчанию»).

Формат каждой строки:

  ALLOW или DENY   адрес_источника   протокол   [порт_источника -> порт_назначения]

Где:

  ALLOW — разрешить соединение
  DENY  — заблокировать соединение

  адрес_источника — откуда пришёл пакет. Варианты:
    *                              любой адрес
    200::1                         конкретный IPv6-адрес
    200::/7                        целая подсеть (все адреса, начинающиеся с 200:: по 3ff::...)
    325:62b8:f811:b821::/64        конкретный сегмент сети

  протокол:
    tcp     — только TCP (веб, SSH и большинство привычных протоколов)
    udp     — только UDP (DNS, игры, видеозвонки...)
    icmpv6  — только ping и служебные сообщения IPv6
    any     — всё вышеперечисленное сразу

  порты (только для tcp, udp, any):
    * -> *     любой порт источника, любой порт назначения
    * -> 443   любой источник, только порт 443 (HTTPS)
    53 -> *    только ответы от DNS-сервера (порт 53) на любой порт


Что такое conntrack и зачем он нужен?
---------------------------------------

Представьте ситуацию: вы открываете браузер и заходите на сайт. Ваш компьютер
отправляет запрос серверу — это исходящее соединение. Сервер отвечает — это
входящий трафик. Но если жёстко заблокировать все входящие соединения, ответ
сервера тоже будет заблокирован, и сайт не откроется.

Conntrack решает эту проблему: программа «запоминает» ваши исходящие соединения
и автоматически разрешает входящие ответы именно от тех адресов и портов, куда
вы только что обращались. Через некоторое время (задаётся через ct_tcp_timeout
и другие параметры) это автоматическое разрешение истекает.

Благодаря conntrack можно смело запрещать весь входящий трафик в правилах —
нужные ответы всё равно будут доходить, а незваные гости — нет.


Примеры файла rules.txt
------------------------

Ниже приведён практический пример настройки. Правила применяются строго сверху
вниз — порядок важен.

  # === Доверенные адреса: пропускаем всё ===

  # Конкретный доверенный хост — разрешить любой трафик
  ALLOW 225:62b8:f811:b821:6c6b:8fcc:c01b:425c  any  * -> *

  # Доверенная подсеть — тоже разрешаем всё
  ALLOW 325:62b8:f811:b821::/64  any  * -> *

  # === DNS-серверы: разрешаем только ответы (входящий UDP, порт источника 53) ===
  ALLOW 308:84:68:55::  udp  53 -> *
  ALLOW 308:25:40:bd::  udp  53 -> *

  # === Разрешить входящие подключения на веб-сервер (если он у вас есть) ===
  ALLOW *  tcp  * -> 80
  ALLOW *  tcp  * -> 443

  # === Разрешить ping (иначе вас не будет «слышно» в сети) ===
  ALLOW *  icmpv6

  # === Заблокировать диапазоны Yggdrasil (идут последними!) ===
  DENY  200::/7   any  * -> *
  DENY  300::/64  any  * -> *


Минимальная конфигурация
-------------------------

Если вам всего лишь необходимо защитить свой компьютер от нежелательных
соединений и не нужно давать доступ для разрешённых адресов, просто добавьте
в rules.txt два правила — этого достаточно:

  DENY  200::/7   any  * -> *
  DENY  300::/64  any  * -> *

Больше ничего настраивать не нужно.
